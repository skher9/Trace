## 1. Feature Understanding
### What This Module Does
Handles all payment processing via Stripe. Accepts a charge request from the orders module, creates a Stripe PaymentIntent, confirms it, and returns success or failure to the caller. Also handles webhook events from Stripe for async 3DS confirmation.

### Files In This Module
- `payments.module.ts`
- `payments.service.ts`
- `payments.controller.ts`
- `payments.dto.ts`
- `stripe.client.ts`
- `payments-trace.md`

### Current Behaviour
1. Order service calls `PaymentsService.charge(orderId, amount, currency)`
2. Service looks up customer's saved payment method from database
3. Creates a Stripe PaymentIntent with `confirm: true`
4. If Stripe returns success → emits `payment.success` event
5. If Stripe returns failure → emits `payment.failed` event and throws
6. Stripe webhook at `/payments/webhook` handles async confirmations for 3DS payments
7. Webhook verifies Stripe signature before processing

### Known Limitations
- Only supports Stripe. No other payment provider.
- No support for multi-currency split payments.
- Refunds must be initiated manually from Stripe dashboard — no refund API yet.

### Recent Changes
- 2026-05-20: Added webhook signature verification (was missing, flagged in security review)
- 2026-04-15: Fixed duplicate charge bug — idempotency key was not being set
- 2026-03-02: Switched from Charges API to PaymentIntents API for 3DS support

### [2026-05-20] [Priya Sharma] — Webhook accepting requests without signature verification
**What Happened:** Security review found our Stripe webhook endpoint was processing all incoming requests without verifying the Stripe-Signature header. Anyone could POST to /payments/webhook and trigger payment.success events.
**Root Cause:** Webhook was set up quickly during initial integration. Stripe signature verification was in the docs but not implemented.
**Fix:** Added `stripe.webhooks.constructEvent()` with the webhook secret from config. Now rejects any request where signature doesn't match.
**Failed Attempts:** Tried verifying in a NestJS guard first, but the raw body was already parsed by then — signature check needs the raw buffer. Had to move verification into the service with `rawBody` from the request.
**Watch Out:** NestJS parses the body before guards run. For webhook signature verification you MUST use the raw body buffer. Add `rawBody: true` to app bootstrap and inject `@RawBody()` in the controller. If you ever change body parsing settings, re-test this.
**Affects:** Security posture of entire payments flow. notifications module was also indirectly at risk since it acts on payment events.

### [2026-04-15] [James Okonkwo] — Duplicate charges on network timeout
**What Happened:** Customers were being charged twice on slow connections. Frontend retry on timeout created a second PaymentIntent and charge.
**Root Cause:** No idempotency key on PaymentIntent creation. Each request created a new PaymentIntent even for the same order.
**Fix:** Now generating idempotency key as `payment-{orderId}-{attemptNumber}`. Stripe deduplicates on their end.
**Failed Attempts:** Tried using orderId alone — blocked legitimate retry after genuine failure (card declined, customer updates card). Added attemptNumber to allow controlled retries.
**Watch Out:** Idempotency keys expire after 24 hours on Stripe. Order retried after 24 hours will create a new PaymentIntent — expected and fine.
**Affects:** orders module was showing duplicate payment.success events before fix.

### [2026-03-02] [Priya Sharma] — Migrated from Charges API to PaymentIntents API
**What Happened:** SCA/3DS compliance required by card networks. Charges API cannot handle 3DS challenges. Some EU customers were seeing blanket declines.
**Root Cause:** Charges API is legacy — does not support the `requires_action` flow for 3DS.
**Fix:** Rewrote payments.service.ts to use PaymentIntents. Added webhook handler for `payment_intent.succeeded` and `payment_intent.payment_failed` for async 3DS confirmation.
**Failed Attempts:** Tried `payment_method_types: ['card']` only — blocked Apple Pay users. Changed to let Stripe determine payment method types automatically.
**Watch Out:** PaymentIntent flow is async for 3DS cards. `charge()` returning does NOT mean payment is captured. Listen to webhook events, not the return value, for final confirmation.
**Affects:** orders module (changed how it waits for payment confirmation), notifications module (updated event names).

## 6. Decision Log
> Every architectural or product decision made in this module.
> Never delete. Always append. Newest at top.

### [2026-05-20] — Webhook signature verification required at service layer, not guard
**Decision:** Stripe webhook signature verification happens inside the service method using the raw request body, not in a NestJS guard.
**Why:** NestJS parses the request body before guards run. Stripe signature verification requires the raw buffer — once parsed, the check fails.
**Alternatives Considered:** Custom NestJS guard; middleware before body parser.
**Tradeoffs Accepted:** Verification logic is not in a reusable guard. Must remember this constraint if the controller is refactored.
**Decided By:** Priya Sharma
**Revisit When:** If NestJS adds a native raw-body-before-guard feature, or if we switch away from Stripe.
**Status:** Active

### [2026-03-02] — Use Stripe PaymentIntents over Charges API
**Decision:** All payment processing uses the PaymentIntents API. The legacy Charges API is fully removed.
**Why:** 3DS/SCA compliance required by European card networks. Charges API cannot handle the `requires_action` async flow.
**Alternatives Considered:** Staying on Charges API with 3DS exemptions; using a different payment provider.
**Tradeoffs Accepted:** More complex async flow — payment confirmation comes via webhook, not return value. More state to manage in orders module.
**Decided By:** Dev team + compliance requirement
**Revisit When:** If Stripe deprecates PaymentIntents (unlikely). Otherwise permanent.
**Status:** Active

- [ ] Check application logs for Stripe error codes
- [ ] Check config service is running — Stripe keys loaded correctly?
- [ ] Check database connection — payment methods lookup failing?
- [ ] Check recent deployments — did something ship in the last hour?

### Common Fixes
| Symptom | Fix | Time to Fix |
|---------|-----|-------------|
| Webhook not firing | Check webhook secret in config matches Stripe dashboard | 5 min |
| Duplicate charges | Check idempotency key generation — orderId and attemptNumber both present | 10 min |

| All payments failing 401 | Auth service down — payments requires valid JWT | 5 min |

- Customer money taken but order not fulfilled
