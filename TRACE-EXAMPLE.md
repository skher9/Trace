## 1. Feature Understanding
### What This Module Does
Handles all payment processing via Stripe. Accepts a charge request from the orders module, creates a Stripe PaymentIntent, confirms it, and returns success or failure to the caller. Also handles webhook events from Stripe for async confirmation.

### Files In This Module
- `payments.module.ts`
- `payments.service.ts`
- `payments.controller.ts`
- `payments.dto.ts`
- `stripe.client.ts`
- `payments-trace.md`

### Current Behaviour
1. Order service calls `PaymentsService.charge(orderId, amount, currency)`
2. Service looks up customer's saved payment method from database
3. Creates a Stripe PaymentIntent with `confirm: true`
4. If Stripe returns success → emits `payment.success` event
5. If Stripe returns failure → emits `payment.failed` event and throws
6. Stripe webhook at `/payments/webhook` handles async confirmations for 3DS payments
7. Webhook verifies Stripe signature before processing

### Known Limitations
- Only supports Stripe. No other payment provider.
- No support for multi-currency split payments.
- Refunds must be initiated manually from Stripe dashboard — no refund API yet.

### Recent Changes
- 2026-05-20: Added webhook signature verification (was missing, flagged in security review)
- 2026-04-15: Fixed duplicate charge bug — idempotency key was not being set
- 2026-03-02: Switched from Charges API to PaymentIntents API for 3DS support

## 3. Expected Behaviour

### ✅ What Should Happen
- Every charge attempt gets a unique idempotency key to prevent duplicate charges
- Stripe webhook signature is verified before any webhook event is processed
- payment.success event fires only after Stripe confirms payment is captured
- payment.failed event fires on any Stripe decline or error
- All payment amounts stored in smallest currency unit (pence, cents) to avoid float errors

### ❌ Intentional Absences — What Will NOT Happen & Why
| What | Why Intentional | Documented By | Date |
|------|----------------|---------------|------|
| No refund API endpoint | Refunds require manual review by finance team before processing. API would bypass this. | Priya Sharma | 2026-03-15 |
| No logging of raw Stripe responses | Raw responses contain card metadata — logging would violate PCI DSS scope | Dev team | 2026-04-01 |
| No retry on payment failure | Retrying a declined card without user action is pointless and risks card lockout | James Okonkwo | 2026-05-01 |

### ⚠️ Conditional Behaviours
| Condition | Behaviour | Reason |
|-----------|-----------|--------|
| 3DS authentication required | PaymentIntent status = `requires_action`, webhook handles final confirmation | Stripe 3DS flow is async |
| Stripe API timeout (>10s) | Throws PaymentTimeoutError, order stays in pending_payment | Safer to leave pending than double-charge |
| Webhook received with invalid signature | Request rejected immediately with 400, no processing | Prevents spoofed webhook attacks |

### [2026-05-20] [Priya Sharma] — Webhook accepting requests without signature verification
**What Happened:** Security review found our Stripe webhook endpoint was processing all incoming requests without verifying the Stripe-Signature header. Anyone could POST to /payments/webhook and trigger payment.success events.
**Root Cause:** Webhook was set up quickly during initial integration. Stripe signature verification was in the docs but not implemented.
**Fix:** Added `stripe.webhooks.constructEvent()` call with the webhook secret from config. Now rejects any request where signature doesn't match.
**Failed Attempts:** Tried verifying in a NestJS guard first, but the raw body was already parsed by the time the guard ran — signature check needs the raw buffer. Had to move verification into the service method with `rawBody` from the request.
**Watch Out:** NestJS parses the body before guards run. For webhook signature verification you MUST use the raw body buffer. Add `rawBody: true` to the NestJS app bootstrap and inject `@RawBody()` in the controller. If you ever change body parsing settings, re-test this.
**Affects:** Security posture of entire payments flow. notifications module was also indirectly at risk since it acts on payment events.

### [2026-04-15] [James Okonkwo] — Duplicate charges on network timeout
**What Happened:** Customers were being charged twice on slow connections. When Stripe request timed out, the frontend retried, creating a second PaymentIntent and charge.
**Root Cause:** No idempotency key was set on PaymentIntent creation. Each request created a new PaymentIntent even for the same order.
**Fix:** Now generating idempotency key as `payment-{orderId}-{attemptNumber}` and passing it in the Stripe API options. Stripe deduplicates on their end.
**Failed Attempts:** Tried using orderId alone as the idempotency key, but this blocked legitimate retry after a genuine failure (e.g. card declined then customer updates card). Added attemptNumber to allow controlled retries.
**Watch Out:** Idempotency keys expire after 24 hours on Stripe. If an order is retried after 24 hours it will create a new PaymentIntent — this is fine and expected.
**Affects:** orders module was showing duplicate `payment.success` events before fix.

### [2026-03-02] [Priya Sharma] — Migrated from Charges API to PaymentIntents API
**What Happened:** SCA/3DS compliance required by card networks. Charges API cannot handle 3DS challenges. Some EU customers were seeing blanket declines.
**Root Cause:** Charges API is legacy and does not support the `requires_action` flow needed for 3DS authentication.
**Fix:** Rewrote payments.service.ts to use PaymentIntents. Added webhook handler for `payment_intent.succeeded` and `payment_intent.payment_failed` events to handle async 3DS confirmation.
**Failed Attempts:** Tried using `payment_method_types: ['card']` only — but this blocked Apple Pay users. Changed to let Stripe determine payment method types automatically.
**Watch Out:** The PaymentIntent flow is async for 3DS cards. The `charge()` method returning does NOT mean payment is captured. Listen to the webhook events, not the return value, for final confirmation. orders module was updated to wait for the payment.success event before marking order complete.
**Affects:** orders module (had to change how it waits for payment confirmation), notifications module (had to update event names it listens to).
