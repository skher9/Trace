## 1. Feature Understanding
### What This Module Does
Handles all payment processing via Stripe. Accepts a charge request from the orders module, creates a Stripe PaymentIntent, confirms it, and returns success or failure to the caller. Also handles webhook events from Stripe for async 3DS confirmation.

### Files In This Module
- `payments.module.ts`
- `payments.service.ts`
- `payments.controller.ts`
- `payments.dto.ts`
- `stripe.client.ts`
- `payments.spec.ts`
- `payments.e2e.spec.ts`
- `payments-trace.md`

### Current Behaviour
1. Order service calls `PaymentsService.charge(orderId, amount, currency)`
2. Service looks up customer's saved payment method from database
3. Creates a Stripe PaymentIntent with `confirm: true`
4. If Stripe returns success → emits `payment.success` event
5. If Stripe returns failure → emits `payment.failed` event and throws
6. Stripe webhook at `/payments/webhook` handles async confirmations for 3DS payments
7. Webhook verifies Stripe signature before processing

### Known Limitations
- Only supports Stripe. No other payment provider.
- No support for multi-currency split payments.
- Refunds must be initiated manually from Stripe dashboard — no refund API yet.

### Recent Changes
- 2026-05-20: Added webhook signature verification (was missing, flagged in security review)
- 2026-04-15: Fixed duplicate charge bug — idempotency key was not being set
- 2026-03-02: Switched from Charges API to PaymentIntents API for 3DS support

## 4. Memory Log
> Never delete entries. Always append. Newest at top.

### [2026-05-20] [Priya Sharma] — Webhook accepting requests without signature verification
**What Happened:** Security review found our Stripe webhook endpoint was processing all incoming requests without verifying the Stripe-Signature header. Anyone could POST to /payments/webhook and trigger payment.success events.
**Root Cause:** Webhook was set up quickly during initial integration. Stripe signature verification was in the docs but not implemented.
**Fix:** Added `stripe.webhooks.constructEvent()` with the webhook secret from config. Now rejects any request where signature doesn't match.
**Failed Attempts:** Tried verifying in a NestJS guard first, but the raw body was already parsed — signature check needs the raw buffer. Moved verification into the service with `rawBody` from the request.
**Watch Out:** NestJS parses the body before guards run. Webhook signature verification MUST use raw body buffer. Add `rawBody: true` to app bootstrap and inject `@RawBody()` in the controller.
**Affects:** Security posture of entire payments flow. notifications module was also indirectly at risk.

### [2026-04-15] [James Okonkwo] — Duplicate charges on network timeout
**What Happened:** Customers charged twice on slow connections. Frontend retry on timeout created a second PaymentIntent.
**Root Cause:** No idempotency key on PaymentIntent creation.
**Fix:** Idempotency key now `payment-{orderId}-{attemptNumber}`. Stripe deduplicates.
**Failed Attempts:** orderId alone blocked legitimate retry after genuine failure. Added attemptNumber.
**Watch Out:** Idempotency keys expire after 24 hours on Stripe. Retry after 24h creates new PaymentIntent — expected and fine.
**Affects:** orders module was showing duplicate payment.success events before fix.

## 6. Decision Log
> Every architectural or product decision made in this module.
> Never delete. Always append. Newest at top.

### [2026-05-20] — Webhook signature verification at service layer, not guard
**Decision:** Stripe webhook signature verification happens inside the service method using raw body, not a NestJS guard.
**Why:** NestJS parses request body before guards run. Stripe verification requires the raw buffer.
**Alternatives Considered:** Custom guard; middleware before body parser.
**Tradeoffs Accepted:** Verification not in a reusable guard. Must remember if controller is refactored.
**Decided By:** Priya Sharma
**Revisit When:** If NestJS adds native raw-body-before-guard support, or if we switch away from Stripe.
**Status:** Active

### [2026-03-02] — Use Stripe PaymentIntents over Charges API
**Decision:** All payment processing uses PaymentIntents API. Charges API fully removed.
**Why:** 3DS/SCA compliance required by European card networks.
**Alternatives Considered:** Staying on Charges API with 3DS exemptions; different payment provider.
**Tradeoffs Accepted:** Async flow — payment confirmation comes via webhook, not return value.
**Decided By:** Dev team + compliance requirement
**Revisit When:** If Stripe deprecates PaymentIntents. Otherwise permanent.
**Status:** Active
